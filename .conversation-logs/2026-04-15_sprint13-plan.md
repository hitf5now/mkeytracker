**Source:** Plan agent audit of the M+ Challenge Platform monorepo

## 1. Current State Audit

### 1.1 Bot entry & command registration

`apps/bot/src/index.ts`
- Single Discord.js v14 client, intents `Guilds` only.
- On `ClientReady`, calls `syncGuildCache()` which writes `client.guilds.cache` (id/name/icon) to Redis key `bot:guilds` as JSON. Re-synced on `GuildCreate` / `GuildDelete`.
- Starts a Redis pub/sub subscriber on `mplus:bot-notifications` (only if `DISCORD_EVENTS_CHANNEL_ID` is set ÔÇö see audit flag below).
- Dispatches slash commands, buttons, select menus, modals by `customId`.

`apps/bot/src/deploy-commands.ts`
- If `DISCORD_GUILD_ID` is set Ôćĺ registers to that one guild (instant, dev-mode).
- If empty Ôćĺ registers **globally** via `Routes.applicationCommands(DISCORD_CLIENT_ID)`.
- Run manually via `npm run deploy-commands`. **There is no automatic per-guild command registration on `GuildCreate`.** Today this is fine because prod uses global commands; however, there's no explicit guild-scoped re-deploy hook.

`apps/bot/src/config/env.ts`
- Env: `DISCORD_BOT_TOKEN`, `DISCORD_CLIENT_ID`, optional `DISCORD_GUILD_ID` (dev), `API_BASE_URL`, `API_INTERNAL_SECRET`, `REDIS_URL`, optional `DISCORD_EVENTS_CHANNEL_ID`.
- `DISCORD_EVENTS_CHANNEL_ID` is the **single hardcoded fallback channel** ÔÇö the bot's original "post events here" pin.

### 1.2 Notifications / Event embed publishing

`apps/bot/src/lib/notifications.ts`
- Subscriber for `event_created` and `event_updated`.
- `handleEventCreated()` resolves target channel in priority order:
  1. `event.discordGuildId` Ôćĺ `apiClient.getGuildConfig(guildId)` Ôćĺ `config.eventsChannelId`
  2. Fallback to `env.DISCORD_EVENTS_CHANNEL_ID`
  3. If none, logs and skips.
- Posts the event embed, then calls `apiClient.storeDiscordMessage(eventId, messageId, channelId)` to persist it for later edits.
- **Startup guard:** `startNotificationSubscriber()` early-returns if `DISCORD_EVENTS_CHANNEL_ID` is unset. Meaning: if a prod deployment removes the legacy env var, **no embeds post at all**, even for multi-guild installs. This is a multi-tenant blocker.
- `handleEventUpdated()` re-renders using `event.discordMessageId` + `event.discordChannelId` stored on the `Event` row. No `discordGuildId` check ÔÇö OK for editing, since the message/channel IDs are already resolved.

### 1.3 Slash commands

`apps/bot/src/commands/`
- `ping`, `register`, `link`, `profile`, `leaderboard`, `event (assign-groups|start|cancel)`, `signup`, `setup (companion|events-channel)`.
- `/setup events-channel` ÔÇö already multi-tenant-aware. Reads `interaction.guildId`, calls `apiClient.setGuildConfig(guildId, {eventsChannelId, guildName})`. Good foundation.
- `/event assign-groups|start|cancel` ÔÇö **NO permission check, NO guild scoping**. Any user in any guild can pass an eventId and transition the event. Security gap for multi-tenant.
- `/signup`, `/register`, `/link`, `/profile`, `/leaderboard` ÔÇö user-centric, guild-agnostic, work fine across tenants.
- `/setup companion` ÔÇö purely informational.

### 1.4 Bot components (button/modal flows)

`apps/bot/src/components/event-signup.ts` handles:
- `event-signup:{eventId}`, `event-tentative:{eventId}`, `event-decline:{eventId}`
- `event-char`, `event-spec`, `event-manual`, `event-confirm`, `event-role` and team equivalents.
- Components **do not check guild membership** ÔÇö a user who sees an embed (or the event ID) in any guild can sign up. In a multi-tenant world: if Guild B somehow surfaces Guild A's event, cross-guild signups are possible. Practically low risk, but worth gating by `event.discordGuildId === interaction.guildId` for integrity.

### 1.5 API: guild config + events

`apps/api/src/routes/guilds.ts`
- `POST /api/v1/guilds/:guildId/config` ÔÇö upsert `eventsChannelId`, `guildName`.
- `GET /api/v1/guilds/:guildId/config` ÔÇö fetch.
- `GET /api/v1/bot/guilds` ÔÇö returns `bot:guilds` from Redis (what the bot is currently in).
- All internal-auth only. No per-user authorization ÔÇö anyone with the internal secret can overwrite any guild's config.

`apps/api/src/routes/events.ts`
- `POST /events` accepts `discordGuildId`. Events already have a `discordGuildId` column.
- `GET /events?guildIds=ÔÇŽ` supports filtering by a list of guilds (used by web for guild-intersection display ÔÇö see below).
- Event creation publishes `mplus:bot-notifications` with `{type: "event_created", eventId}` (no guild info in payload ÔÇö bot rehydrates).

`apps/api/src/lib/discord-webhook.ts` + `routes/runs.ts`
- On run submission, fires a webhook to **one global** URL: `env.DISCORD_WEBHOOK_RESULTS`. This is single-tenant. Multi-tenant will need a per-guild results channel/webhook.

### 1.6 Schema

`apps/api/prisma/schema.prisma`
- `GuildConfig { discordGuildId @unique, eventsChannelId, guildName }` ÔÇö already exists (good starting point).
- `Event { discordGuildId, discordMessageId, discordChannelId }` ÔÇö event is nullable-guild-scoped.
- `User { discordId @unique, ... }` ÔÇö a User has **no relationship to any guild**. A user is a global identity, not a tenant member.
- No `GuildMember` table, no `GuildAdmin` table, no `Team.guildId`.
- Migrations: 15 sprint migrations to date, last: `20260413250000_rename_points_to_juice`.

### 1.7 Web app

`apps/web/src/lib/auth.ts`
- NextAuth v5, Discord provider, scopes `"identify guilds"`. No `guilds.members.read` and no bot install/OAuth flow.
- On sign-in, JWT stores `discordAccessToken` for server-side guild lookups; calls API `POST /api/v1/auth/discord-login` to upsert the User row.

`apps/web/src/app/api/guilds/route.ts`
- Fetches the user's guilds from Discord REST API (`users/@me/guilds`).
- Intersects against `bot:guilds` from Redis (via internal API).
- Returns the shared list ÔÇö used by `EventCreateForm` server selector.

`apps/web/src/components/event-create-form.tsx`
- Passes `discordGuildId` to event create POST. So events already can be guild-scoped from the web ÔÇö the piece that's missing is guild-scoped reading/filtering across most views.

### 1.8 What's hardcoded vs configurable (summary)

| Concern | Today | Multi-tenant needs |
|---|---|---|
| Events channel | Per-guild in `GuildConfig`, legacy env fallback | Per-guild only; drop env fallback |
| Results channel (run completed) | Single `DISCORD_WEBHOOK_RESULTS` env | Per-guild column + optional webhook or bot send |
| Signup channel/embed | Same as events channel | Probably same, but split is possible |
| Slash commands | Global registration | Keep global, but gate admin commands by Discord permissions |
| Event ownership | `createdByUserId` only | Add `discordGuildId` gate on all lifecycle actions |
| "What servers do I see?" | Bot+user intersection via Redis | Works today, but needs /servers page + admin scoping |
| Bot join/leave | Sync guild cache only | Must create/remove `Guild` row, send welcome DM, re-deploy commands |
| Users Ôćö Guilds | No relationship | New table, or derive on demand from Discord |
| Permissions | None | Use Discord `MANAGE_GUILD` permission bit for admin commands |

## 2. Multi-Server Architecture Proposal

### 2.1 Schema changes (new Prisma migration: `20260415xxxxxx_multi_tenant`)

**Promote `GuildConfig` Ôćĺ `Guild` (first-class tenant record).** Recommended: new `Guild` model, migrate data in.

```prisma
model Guild {
  id                  Int      @id @default(autoincrement())
  discordGuildId      String   @unique @map("discord_guild_id")
  guildName           String?  @map("guild_name")
  guildIconUrl        String?  @map("guild_icon_url")
  /// Snowflake of the admin who installed / initialized the bot
  installedByDiscordId String? @map("installed_by_discord_id")
  /// Bot currently member? Set false on GuildDelete to soft-retain history
  botActive           Boolean  @default(true) @map("bot_active")

  // Channel routing
  eventsChannelId     String?  @map("events_channel_id")
  resultsChannelId    String?  @map("results_channel_id")
  announcementsChannelId String? @map("announcements_channel_id")
  /// Optional per-guild webhook URL for run results (if they want decoupled delivery)
  resultsWebhookUrl   String?  @map("results_webhook_url")

  // Feature toggles
  allowPublicEvents   Boolean  @default(true) @map("allow_public_events")
  timezone            String?  // IANA, defaults to event-creator's tz

  createdAt           DateTime @default(now()) @map("created_at")
  updatedAt           DateTime @updatedAt @map("updated_at")

  admins   GuildAdmin[]
  members  GuildMember[]
  events   Event[]
  teams    Team[]

  @@map("guilds")
}

model GuildAdmin {
  id              Int      @id @default(autoincrement())
  guildId         Int      @map("guild_id")
  guild           Guild    @relation(fields: [guildId], references: [id], onDelete: Cascade)
  userId          Int      @map("user_id")
  user            User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  /// "owner" | "admin" ÔÇö derived from Discord MANAGE_GUILD bit at time of verification
  role            String   @default("admin")
  /// Last time we verified via Discord API that this user still has MANAGE_GUILD
  verifiedAt      DateTime @default(now()) @map("verified_at")

  @@unique([guildId, userId])
  @@map("guild_admins")
}

model GuildMember {
  id        Int      @id @default(autoincrement())
  guildId   Int      @map("guild_id")
  guild     Guild    @relation(fields: [guildId], references: [id], onDelete: Cascade)
  userId    Int      @map("user_id")
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  /// User's chosen default guild for publishing results (one Primary per user)
  isPrimary Boolean  @default(false) @map("is_primary")
  joinedAt  DateTime @default(now()) @map("joined_at")

  @@unique([guildId, userId])
  @@index([userId, isPrimary])
  @@map("guild_members")
}
```

**Changes to existing models:**
- `User` ÔÇö add `guildMemberships GuildMember[]`, `guildAdminships GuildAdmin[]`.
- `Event` ÔÇö change `discordGuildId String?` to `guildId Int?` FK to `Guild`. Decision: **fully normalize** (`guildId Int`, required once we migrate).
- `Team` ÔÇö add `guildId Int?` (optional ÔÇö teams may be personal / cross-guild, or scoped). Recommend required per-guild to match Discord-server community model.
- `Run` ÔÇö NO guild column needed. The **announcement target** is computed from the owning user's Primary Guild at post time.
- Migrate existing `guild_configs` rows into `guilds`; backfill `events.guild_id` from `events.discord_guild_id` when present. Events with null guild become "orphaned" ÔÇö either assign to a designated "default guild" during migration or leave null.

### 2.2 API changes

New/modified routes (all under `/api/v1`):

- `POST /guilds/:guildId/init` ÔÇö called by bot on `GuildCreate` event.
- `POST /guilds/:guildId/uninstall` ÔÇö called on `GuildDelete`. Flips `botActive = false`.
- `GET /guilds/:guildId` ÔÇö guild detail (for web admin UI). Requires admin.
- `PATCH /guilds/:guildId/config` ÔÇö update channel mappings, webhooks, toggles. Requires admin.
- `POST /guilds/:guildId/admins/verify` ÔÇö verify-and-upsert `GuildAdmin` for current session user via Discord API.
- `GET /guilds/:guildId/admins` ÔÇö list verified admins.
- `POST /users/me/primary-guild` ÔÇö set current user's default guild for result publishing.
- `GET /users/me/guilds` ÔÇö list `GuildMember` rows for current user.
- **Authorization middleware:** new `requireGuildAdmin(guildId)` Fastify plugin with TTL re-verification (24h).
- `POST /events` ÔÇö require `discordGuildId`, require `GuildMember`. Wide-open to members for create.
- `POST /events/:id/transition`, `assign-groups`, `close-signups` ÔÇö require admin OR event creator.
- Results webhook: replace `env.DISCORD_WEBHOOK_RESULTS` with per-user Ôćĺ Primary Guild Ôćĺ `guild.resultsChannelId` (bot send via token).

### 2.3 Bot changes

- **Slash command registration:** stay **global**.
- **Admin-gated commands**: use `.setDefaultMemberPermissions(PermissionFlagsBits.ManageGuild)` AND runtime re-check.
- **`GuildCreate` handler:** POST to `/api/v1/guilds/:guildId/init`, send welcome embed, DM the inviter.
- **`GuildDelete` handler:** POST to `/api/v1/guilds/:guildId/uninstall`.
- **Remove hardcoded `DISCORD_EVENTS_CHANNEL_ID` fallback** in `notifications.ts`.
- **New `/setup` subcommands:** `results-channel`, `show`, `web` (DMs tokenized link to web admin UI).
- **Gate component handlers by guild:** compare `interaction.guildId` to `Event.discordGuildId`.

### 2.4 Onboarding flow

1. Admin clicks **"Add Bot to Server"** on the web landing page Ôćĺ Discord OAuth (`scope=bot applications.commands`, permissions bitmask).
2. Admin selects server, authorizes. Discord sends `GuildCreate`.
3. Bot POSTs `/guilds/:guildId/init` Ôćĺ creates `Guild` row.
4. Bot DMs inviter with setup link / `/setup` command hint.
5. Admin lands on `/servers/<guildId>`, picks channels from dropdowns (channels populated via bot API).
6. Writes `GuildConfig`. Done.

### 2.5 Permissions model

1. **Bot** ÔÇö internal secret, full power.
2. **Guild Admin** ÔÇö `MANAGE_GUILD` Discord perm; cached with TTL re-verify.
3. **Guild Member** ÔÇö any Discord user in the guild.

## 4. User Ôćĺ Discord Server Binding

Three layers, in decreasing precedence:

1. **Per-event explicit binding** (highest): `event.guildId` Ôćĺ result posts to that event's guild.
2. **Per-team binding**: `team.guildId` Ôćĺ team runs and team event results route here.
3. **Per-user Primary Guild** (fallback): `GuildMember.isPrimary = true` for ad-hoc runs.
4. **If no Primary**: no broadcast. User can set anytime.

Edge cases:
- Multi-server user, no primary set Ôćĺ first run prompts via DM with select menu.
- User leaves a guild Ôćĺ `isPrimary` invalidates; re-prompt.
- User opts out entirely Ôćĺ `User.broadcastRuns: Boolean = false`.

Team runs: derived from `Run.groupId Ôćĺ EventGroup.eventId Ôćĺ Event.guildId`, or from `team.guildId` for raw team runs.

## 5. Interview Questions

1. **Existing events on migration:** backfill all to your home server, leave null and hide, or delete?
2. **Cross-guild events:** 1:1 (one event, one guild) or N:N (one event, multiple guild embeds)?
3. **Team scoping:** guild-scoped, global, or optionally guild-scoped?
4. **Run broadcast default:** auto-broadcast to Primary Guild, or opt-in via `User.broadcastRuns` (recommended)?
5. **Bot hosting:** stay single shared instance, or support self-hosting?
6. **Bot OAuth scopes:** minimum is `bot + applications.commands`. Want Manage Webhooks (auto-create result webhooks)? Manage Roles (auto-assign "Registered" role)?
7. **Channel list retrieval:** bot internal HTTP endpoint, API uses bot token directly (recommended), or front-end uses user's Discord token?
8. **Admin verification TTL:** 24h + on-demand recheck? Or every action?
9. **Results webhook vs bot send:** webhook URL per guild (decoupled, admin must create) or bot sends via token (no extra admin step, recommended)?
10. **Bot removal handling:** soft-flag `botActive=false` with 90-day purge of configs, keep events/runs forever ÔÇö confirm window?
11. **`/servers` for non-admin members:** show event history per shared guild? (Low priority, nice-to-have.)
12. **Legacy env vars:** drop `DISCORD_EVENTS_CHANNEL_ID`, `DISCORD_WEBHOOK_RESULTS`, `DISCORD_WEBHOOK_EVENTS` in Sprint 13, or keep as emergency fallback?
13. **Unraid prod impact:** OK with ~5s bot restart for `notifications.ts` subscriber fix?
14. **Naming:** Prisma model `Guild` (Discord) vs `DiscordServer` / `Tenant` to avoid WoW guild ambiguity?
15. **Scale target:** 1, 10, 100, or 1000 servers? Affects caching.
16. **Primary Guild UX:** `/setup primary` slash command + web, or web-only?
17. **Cross-guild visibility on web:** UserA in GuildA signs up; UserB (only in GuildB) ÔÇö web shows all, Discord embed per-guild. Confirm.
18. **Juice pools (Sprint 12):** Personal Juice = global, Event/Team Juice = inherit from event/team guild?
